## Why

Hotspot Foundation 04 — Materialize hotspot artifacts and SQLite E2E verification

The artifact writing code in `crates/scryrs-cli/src/lib.rs` (`write_hotspots_json` and `write_empty_success_report`) already produces stdout and `.scryrs/hotspots.json` for every successful `scryrs hotspots <PATH>` invocation. However, this behavior is **not proven through the public CLI pipeline**: all existing hotspot tests call `populate_store()` which bypasses the `scryrs record` command, and all existing record tests use a private `store_override` that `scryrs hotspots` cannot discover (it resolves `.scryrs/scryrs.db` relative to the `<PATH>` argument, not the override). This means there is currently **zero coverage** of the CWD-based canonical path flow that real users experience.

Additionally, the existing `insta` snapshot infrastructure (already configured in `Cargo.toml` with JSON support) is unused for hotspot outputs — the `snapshots/` directory contains only two help-output snapshots. Without snapshot-based drift detection, accidental contract changes to hotspot stdout or artifact JSON can slip through unnoticed, breaking downstream graph, proposal, and adapter consumers.

Finally, existing hotspot test fixtures use only `FileOpened` events, meaning the scoring weights for `SearchRun`, `SymbolInspected`, `CommandExecuted`, `DocRetrieved`, `EditMade`, `FailedLookup`, and the `+2` failure bonus are verified only through unit tests in `scoring.rs`, not through the CLI's integration surface.

This task closes these verification gaps: it proves the public `record → SQLite → hotspots → hotspots.json` loop end-to-end, adds `insta` snapshots for hotspot output drift detection, and extends fixture coverage to all required subject-bearing event families.

## What Changes

- **Add a dedicated end-to-end integration test binary** at `crates/scryrs-cli/tests/hotspot_e2e.rs` that:
  - Creates a temp repository working directory with CWD serialization
  - Pipes a programmatic multi-event JSONL fixture through `scryrs record --stdin` to populate the canonical `.scryrs/scryrs.db`
  - Verifies `trace_events` rows exist in SQLite via `rusqlite`
  - Runs `scryrs hotspots <repo>` and asserts exit code 0, expected subject ranking, stdout/artifact byte-for-byte equality (modulo stdout trailing newline), and `insta` snapshot assertions with volatile-field normalization
- **Extend existing `hotspot_integration_tests`** in `crates/scryrs-cli/src/lib.rs` to use the full multi-event-family fixture (covering file, search, symbol, command, document, edit, and failure events) for inline ranking and score assertions
- **Add `insta` snapshot assertions** (both `assert_snapshot!` for normalized stdout text and `assert_json_snapshot!` for normalized artifact JSON) with volatile-field normalization: `generatedAt` replaced with `"<GENERATED_AT>"`, `repositoryPath` replaced with `"<REPO>"`, `storePath` replaced with `"<STORE>"`
- **Zero production code changes** — this is pure verification hardening
- Existing targeted unit tests for corrupt, unsupported, and artifact-write-failure error paths remain unchanged

## Impact

- **Affected specs**: `openspec/specs/hotspot-report/spec.md` (no changes needed — spec already defines the contract being verified)
- **Affected code**:
  - `crates/scryrs-cli/tests/hotspot_e2e.rs` (new file — E2E integration test binary)
  - `crates/scryrs-cli/src/lib.rs` (additive changes to `hotspot_integration_tests` module only)
  - `crates/scryrs-cli/src/snapshots/` (new snapshot files generated by `insta`)
  - `crates/scryrs-cli/Cargo.toml` (no changes — `insta`, `rusqlite`, `tempfile` already in dev-dependencies)
- **No impact on**: Production code, APIs, schemas, CLI surface, graph, proposal, adapter, runtime, or LLM code paths
