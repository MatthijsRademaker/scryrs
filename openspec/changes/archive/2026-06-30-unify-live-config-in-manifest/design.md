## Context

Live-mode configuration is split across two files that `scryrs init` keeps in sync by writing the same values to both:

- `scryrs.json` `remote` — committed, holds `ingest_url`, `repository_id`, `workspace_id`, `agent_id`, `timeout_ms`.
- `.scryrs/.env` — gitignored, holds the same four identity fields **plus** the required `SCRYRS_DOCKER_NETWORK`.

The hook resolves transport config every invocation via `resolve_remote_config` with the precedence chain `CLI flag > process env > .scryrs/.env > scryrs.json remote`. So `.env` already wins over the manifest, and the manifest is already the lowest layer. The problems are: (1) `init` duplicates the four identity fields into both files, which can drift and is babysat by `validate_live_bootstrap_state`'s `.env` conflict check; (2) `docker_network` lives only in the gitignored `.env`, so a fresh clone of a shared project has no committed network value; (3) because all managed values live in the gitignored file, a fresh clone silently resolves to local mode.

The prior change `workspace-live-server-bootstrap` (decision #3) deliberately kept `docker_network` out of `scryrs.json` on the grounds that the manifest is "transport identity, not container-orchestration metadata." This change supersedes that decision: we make `scryrs.json` the single committed source of truth for **all** live config, including orchestration.

Constraints:

- The hook's resolution precedence must not change — `.env` and env vars must still override the manifest for local/per-machine needs.
- Existing populated `.scryrs/.env` files must keep working as overrides (backward compatible at read time).
- `scryrs up` and the generated `compose.yml` reference `${SCRYRS_DOCKER_NETWORK}`; that runtime requirement is unchanged — only where the value originates moves.
- Live init must stay fail-fast and all-or-nothing.

## Goals / Non-Goals

**Goals:**

- Make `scryrs.json` the single committed source of truth for the shared live constants (`ingest_url`, `workspace_id`, `docker_network`).
- Stop committing `repository_id` (derive from Git) and `agent_id` (autogenerate per container) — committed values for those are redundant or meaningless.
- Demote `.scryrs/.env` to optional, local-only overrides; `init` stops pre-populating it with managed values.
- Eliminate the identity duplication and the `.env` semantic-conflict guard that only existed to manage drift.
- Keep the hook's runtime resolution precedence and transport logic byte-for-byte unchanged.
- Make a fresh `git clone` of a live-configured project resolve to live mode without re-running `init`.

**Non-Goals:**

- No change to the hook transport branch (`hook.rs`) or the local-vs-remote decision rule (non-empty `ingest_url` ⇒ remote).
- No change to ingest protocol, envelope shape, or server packaging.
- No removal of the override layers — env vars and `.env` remain valid override sources.
- No migration tooling for already-populated `.env` files; they continue to work as overrides.
- No change to local mode.

## Decisions

### 1. `docker_network` becomes a flat field on the existing `remote` object

`scryrs.json` `remote` gains one field:

```json
{
  "remote": {
    "ingest_url": "http://scryrs:8081",
    "workspace_id": "...",
    "docker_network": "agent-network"
  }
}
```

All committed live config lives under a single `remote` object — there is no separate orchestration section. `repository_id` and `agent_id` are intentionally absent (derived / autogenerated — see decision #5). This keeps the committed manifest flat and the resolver simple: one object to read, one place to look. The `remote` object still *accepts* `repository_id`/`agent_id`/`timeout_ms` as optional overrides; `init` just no longer writes them.

**Why flat rather than a separate `live` object:** a sibling object would add a second section and a second reader for one field, and force `scryrs up` to merge two sources. A flat field keeps the manifest shape minimal and the read path single. The conceptual distinction between transport identity and orchestration is documented in prose, not enforced by JSON structure.

**Trade-off accepted:** `remote.*` no longer maps 1:1 to `SCRYRS_REMOTE_*` env vars — `docker_network` maps to `SCRYRS_DOCKER_NETWORK` (no `REMOTE_` prefix). The `scryrs-manifest` spec is updated to describe `remote` as the live-config object (identity + network) rather than transport-ingest-only, and to note the one field whose override env var omits the `REMOTE_` prefix.

**Alternative rejected:** A separate top-level `live` object. Cleaner conceptual separation, but adds structure and a second reader for a single field. Also rejected: keep `docker_network` only in `.env` (option C2) — leaves a required value uncommitted, contradicting the single-source-of-truth goal.

### 2. Resolution gains a manifest layer for the network; precedence is otherwise unchanged

`resolve_docker_network` currently reads `CLI flag > env > .scryrs/.env`. It gains `scryrs.json` `remote.docker_network` as the lowest (committed-base) layer:

```
CLI flag  >  SCRYRS_DOCKER_NETWORK env  >  .scryrs/.env  >  scryrs.json remote.docker_network
```

This mirrors the identity precedence chain exactly (committed manifest is the base; `.env`/env override). The existing manifest reader (`discover_manifest_remote`) is extended to parse `docker_network` from the `remote` object — no new reader needed.

**Why:** Symmetry with the existing identity resolution keeps one mental model — manifest is the committed base, everything else overrides.

### 3. `init` writes all managed config to `scryrs.json`; `.env` is scaffolded as overrides-only

In live mode `init` will:

- Create-or-merge the `scryrs.json` `remote` object with the existing identity fields **and** the new `docker_network` field.
- Scaffold `.scryrs/.env` as an empty or commented stub documenting the overridable keys — it no longer writes managed identity or network values.
- Drop `validate_live_bootstrap_state`'s `.env` managed-value conflict check (nothing managed is written there anymore). The `scryrs.json` manifest conflict check (`validate_live_manifest_state`) is extended to cover `remote.docker_network`.
- The `compose.yml` conflict check is retained as-is (the template is unchanged).

**Why:** Removes the duplication at its source. The conflict guard existed only to keep the two copies consistent; with one source it is dead weight.

**Alternative rejected:** Keep dual-writing but mark `.env` as derived. Still two copies, still drift surface.

### 4. `scryrs up` resolves the network from the manifest and injects it into Compose

The generated `compose.yml` still declares `external: true` / `name: ${SCRYRS_DOCKER_NETWORK}`. Because `init` no longer writes `SCRYRS_DOCKER_NETWORK` into `.env`, `scryrs up` must resolve the network value (via the precedence chain in decision #2) and pass it into the Compose process environment (e.g. set `SCRYRS_DOCKER_NETWORK` in the child env) so variable substitution still resolves.

**Why:** Compose variable substitution needs the value at launch time; sourcing it from the manifest keeps `compose.yml` itself untouched and declarative. `up` already validates the scaffold and the external network; resolving the name is a natural extension.

**Alternative rejected:** Have `up` regenerate `.env` from the manifest before launching. That re-introduces a derived `.env` artifact and the drift surface this change removes.

### 5. The committed manifest holds only shared constants; `repository_id` derives, `agent_id` autogenerates

`scryrs init` writes only the genuinely-shared constants into `remote`: `ingest_url`, `workspace_id`, `docker_network`. The other two former-required fields are removed from committed config:

- **`repository_id`** — derived from `git remote get-url origin` (normalized, `.git`-stripped). `resolve_remote_config` already has this fallback (remote_config.rs:182); init simply stops writing it. An explicit value (flag / env / `.env` / manifest) still wins.
- **`agent_id`** — autogenerated at runtime from a stable per-container identity (the container hostname, e.g. `HOSTNAME`) when not overridden. Today resolution *errors* when `agent_id` is empty (remote_config.rs:200); that hard error is replaced by the autogen fallback.

**Why autogen from the hostname, not a random UUID or the pid:** the hook runs as a *fresh process per tool call* (`PreToolUse` re-invokes `scryrs hook claude-code` each time), so any per-process value (`std::process::id()`, a fresh random UUID) would scatter one session's events across many `agent_id`s and break grouping. The container hostname is stable for the container's whole lifetime and, in Docker, defaults to the container id — meaningful and distinct per ephemeral agent. A fixed committed UUID is the opposite failure: identical across every cloned container, so it identifies nothing.

**Why now (folded into this change):** committing `repository_id`/`agent_id` was already the weakest part of the two-file design — `repository_id` was redundant with Git, and a committed `agent_id` is actively wrong for ephemeral containers. Removing them is what makes "the committed manifest is shared constants only" actually true.

**Alternative rejected:** keep `agent_id` required and committed. Forces every container clone to share an identity or override it, making the committed value a near-always-dead default.

### 6. Hook transport path is untouched

`hook.rs` continues to call `resolve_remote_config(Some(base_dir))` and branch on a resolved non-empty `ingest_url`. Since the manifest is already the base layer and `.env` is already an override, removing managed values from `.env` simply means the manifest now supplies them directly. No code change in `hook.rs`.

**Why:** The whole point of the existing precedence design is that the *source* of a value can move without the consumer caring. This change exploits that.

## Risks / Trade-offs

- [Risk] `docker_network` is genuinely per-machine in some setups, and committing it to `scryrs.json` bakes one team's topology into the repo. → Mitigation: the override chain (env / `.env`) still lets any operator override the committed default locally; the manifest value is a default, not a lock.
- [Risk] Reversing prior decision #3 changes the `scryrs-manifest` scope from "transport identity only" to "live config incl. orchestration," and `remote.*` no longer maps 1:1 to `SCRYRS_REMOTE_*` env vars (`docker_network` → `SCRYRS_DOCKER_NETWORK`). → Mitigation: update the manifest spec + docs to describe `remote` as the live-config object and call out the one field whose override env var omits the `REMOTE_` prefix.
- [Risk] Operators with existing populated `.scryrs/.env` files expect those values to apply. → Mitigation: precedence is unchanged, so existing `.env` values still override the manifest exactly as before — fully backward compatible at read time.
- [Risk] `scryrs up` failing to inject `SCRYRS_DOCKER_NETWORK` would make Compose substitution fail with an opaque error. → Mitigation: `up` resolves the network deterministically and fails loudly with remediation guidance when no value resolves from any layer, before invoking Compose.
- [Risk] A fresh clone now resolves to live mode automatically, which could surprise a developer who expected local. → Mitigation: this is the intended behavior of "committed source of truth"; document it, and `--mode local` plus `.env`/env overrides remain available.
- [Risk] `agent_id` autogen from the hostname is unstable if the harness runs across multiple hosts/containers under one logical session, or the hostname is empty in a bare environment. → Mitigation: keep the explicit override chain (flag / env / `.env`) authoritative so any deployment needing a specific stable id can set one; fall back to a process-derived value only if the hostname is unavailable, and document that committed/overridden `agent_id` is the path for non-container deployments.
- [Risk] Git-derived `repository_id` is empty outside a Git checkout (or when origin is unset), which previously surfaced as a hard error. → Mitigation: resolution still errors loudly with remediation when `repository_id` cannot be derived *and* is not overridden — derivation is a convenience, not a silent default.

## Migration Plan

1. Extend `discover_manifest_remote` to parse `remote.docker_network` and add it to `resolve_docker_network`'s precedence chain as the base layer.
2. Add the `agent_id` autogen fallback (stable container hostname) to `resolve_remote_config`, replacing the `MissingIdentity { field: "agent_id" }` hard error; confirm the existing `repository_id` Git-derive fallback path.
3. Update `init` live mode to write only `ingest_url`/`workspace_id`/`remote.docker_network` into `scryrs.json`, stop writing `repository_id`/`agent_id` and all `.scryrs/.env` managed values, make `agent_id` an optional input, and scaffold `.env` as an overrides-only stub.
4. Extend `validate_live_manifest_state` to cover `remote.docker_network`; remove the `.scryrs/.env` managed-value conflict check.
5. Update `scryrs up` to resolve the network from the manifest precedence chain and inject `SCRYRS_DOCKER_NETWORK` into the Compose environment; keep deterministic fail-fast when unresolved.
6. Update next-step / help text and docs (`live-server-setup.md`, `cli-v0-contract.md`) to describe `scryrs.json` as the committed source of truth, the derived/autogenerated identity, and `.scryrs/.env` as local overrides.
7. Update tests and snapshots: `init_tests.rs`, `up_tests.rs`, `remote_config.rs`, smoke/help-json snapshots.

**Rollback:** Revert to dual-writing identity into `.env`, writing `docker_network` only to `.env`, and committing `repository_id`/`agent_id`, restoring the `.env` conflict check and the `agent_id` hard error. Because `.env` reads are backward compatible and the manifest `remote.docker_network` field is additive, rollback requires no data migration — a stale committed `remote.docker_network` is simply ignored by the older resolver.

## Open Questions

- Exact `.env` stub form: empty file vs. a commented `.env.example`-style stub listing overridable keys. Leaning toward a commented stub for discoverability.
- The precise autogen source order for `agent_id` (e.g. `HOSTNAME` env → `hostname` syscall → process-derived). Hostname-first is the leading approach; the exact fallback ladder can be settled in implementation.
