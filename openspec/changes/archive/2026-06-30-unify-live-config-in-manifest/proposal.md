## Why

Live-mode config is duplicated across two files. `scryrs init` writes the same four identity fields into both `scryrs.json` `remote` and the gitignored `.scryrs/.env`, while the required `docker_network` lives **only** in `.env`. The duplication can drift (guarded today by a brittle conflict-check), and because `.env` is gitignored, a fresh `git clone` of a shared project carries no committed live config — the hook silently falls into local mode. We want `scryrs.json` to be the single committed source of truth for live configuration, with `.scryrs/.env` demoted to optional local overrides only.

Two of the four committed identity fields don't belong in committed config at all: `repository_id` is derivable from the Git remote, and `agent_id` is meaningless as a fixed value in an ephemeral-container setup — every cloned container would share one baked-in UUID. So the committed manifest should hold only the genuinely-shared constants (`ingest_url`, `workspace_id`, `docker_network`); `repository_id` should be derived and `agent_id` autogenerated at runtime.

## What Changes

- **BREAKING (workspace contract):** `scryrs init` (live mode) writes the committed live config — `ingest_url`, `workspace_id`, and `docker_network` — into `scryrs.json` `remote` only. It no longer pre-populates `.scryrs/.env` with managed values.
- `repository_id` is no longer written to the committed manifest; it is derived from the Git remote origin at resolution time (the fallback already exists in `resolve_remote_config`). An explicit override still wins.
- `agent_id` is no longer written to the committed manifest; it is autogenerated at runtime from a stable per-container identity (the container hostname) when not overridden. Fixed committed UUIDs are removed because they are meaningless across ephemeral containers.
- Add a manifest home for the Docker network name as a flat field on the existing `remote` object: `remote.docker_network`. This supersedes prior decision #3 of `workspace-live-server-bootstrap`, which kept the network out of `scryrs.json`.
- `docker_network` resolution gains `scryrs.json` `remote.docker_network` as its committed base layer, below the existing `CLI flag > env > .scryrs/.env` overrides.
- `scryrs up` resolves `docker_network` from the manifest (with override layering) and injects it into the Compose invocation, instead of requiring it to be present in `.scryrs/.env`.
- `.scryrs/.env` becomes overrides-only: `init` creates it empty (or a commented `.env.example`-style stub documenting the override keys) and is gitignored as today. The hook's existing precedence chain (`CLI > env > .scryrs/.env > scryrs.json`) is unchanged, so `.env` still wins when present.
- Remove the managed-value duplication and the `.scryrs/.env` semantic-conflict guard that only existed to babysit drift between the two files.
- Update docs (`live-server-setup.md`, `cli-v0-contract.md`) and `init` next-step / help text so `scryrs.json` is described as the committed source of truth and `.scryrs/.env` as local overrides.

### Modified Capabilities
- `scryrs-manifest`: the `remote` object gains a flat `docker_network` field, drops `repository_id`/`agent_id` from committed defaults (resolution derives/autogenerates them), and becomes the authoritative committed base; the override-precedence requirement extends to the network field.
- `init-installer`: live-mode init writes only the committed shared config (`ingest_url`, `workspace_id`, `docker_network`) into `scryrs.json`, no longer requires `agent_id` as input, and stops pre-populating `.scryrs/.env`; the managed-`.env` conflict requirement is removed and replaced by overrides-only `.env` scaffolding.
- `workspace-live-bootstrap`: `docker_network` source of truth moves from `.scryrs/.env` to `scryrs.json` `remote.docker_network`; `scryrs up` resolves the network from the manifest and injects it into Compose; the scaffold no longer depends on managed identity/network values living in `.env`.

## Impact

- **Code:** `crates/scryrs-cli/src/init.rs` (drop `.env` managed writes + `validate_live_bootstrap_state` env/compose conflict checks; write only `ingest_url`/`workspace_id`/`docker_network` to manifest; stop requiring `agent_id` input), `crates/scryrs-cli/src/remote_config.rs` (extend manifest reader + `resolve_docker_network` to read `remote.docker_network`; add `agent_id` autogen fallback from container hostname; `repository_id` git-derive fallback already present), `crates/scryrs-cli/src/up.rs` and `live_bootstrap.rs` (resolve network from manifest, inject `SCRYRS_DOCKER_NETWORK` into the Compose environment).
- **Hook resolution:** unchanged precedence chain; only the *defaults source* moves. No change to `hook.rs` transport logic.
- **Config files:** `scryrs.json` `remote` grows a `docker_network` field and stops carrying committed `repository_id`/`agent_id`; `.scryrs/.env` no longer carries managed values (existing populated `.env` files — including ones with `repository_id`/`agent_id` — remain valid as overrides, backward compatible at read time).
- **Docs:** `.devagent/docs/docs/live-server-setup.md`, `.devagent/docs/docs/cli-v0-contract.md`, init help/next-step text.
- **Tests:** `init_tests.rs`, `up_tests.rs`, `remote_config.rs` tests, smoke/help snapshots.
