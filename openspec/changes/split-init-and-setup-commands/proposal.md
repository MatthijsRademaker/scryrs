## Why

`scryrs init` conflates two unrelated concerns: installing the harness transport hook (mode-independent, needs no config, every consumer always needs it) and resolving/writing runtime ingest config (`scryrs.json`, `.scryrs/` scaffold). Because `init` defaults to live mode and validates ingest config before doing anything, the one action you *always* need — installing the hook — is gated behind config that is genuinely a runtime concern. Containerized, env-driven deployments (e.g. the Swarm worker image) are forced to pick `--mode local` purely to extract the hook without triggering live-config validation, which is misleading trickery: it implies local recording when the runtime actually records remotely via env. The command surface should make the hook the primitive and config an explicit, separate step.

## What Changes

- **BREAKING**: `scryrs init --agent <name>` becomes **hook-install only**. It installs the harness transport hook, is idempotent, requires no ingest config, and cannot fail on missing/unresolvable live config. `--mode`, `--ingest-url`, `--workspace-id`, `--docker-network`, `--repository-id`, `--agent-id`, and all live/local config-writing behavior are removed from `init`.
- **New command `scryrs setup <mode>`** (`local` | `live`) that owns all runtime-config scaffolding:
  - `setup local` → `.scryrs/` SQLite store + `.scryrs/.gitignore`.
  - `setup live` → creates/merges `scryrs.json` `remote` section (the committed source of truth), scaffolds `.scryrs/`. Fail-fast (exit 2) on missing/invalid required config lives here, where config resolution is the command's actual job.
- **Decouple `docker_network` / compose scaffolding from core `setup live`.** `setup live` requires only `ingest_url` + `workspace_id` (with `repository_id` derived from git origin and `agent_id` autogenerated at runtime, unchanged). The `.scryrs/compose.yml` + `docker_network` scaffolding for a self-hosted live server becomes an explicit opt-in (e.g. `setup live --with-compose` / the `scryrs up` bootstrap), not a hard requirement — deployments that join an existing network or point at an external server never need it.
- **Retarget the interactive wizard to `setup live`.** The in-flight `add-init-interactive-wizard` change adds a TTY wizard for missing live config to `scryrs init`; with the split, that config-collection concern belongs on `scryrs setup live`. The wizard code/UX from that change migrates to `setup`, and `init` carries no interactive prompting (it has no config to collect).
- Update help / help-json, README/CLI docs, golden snapshots, and tests for the new `init` (hook-only) and `setup` surfaces.
- No fail-fast probe work is needed: `scryrs doctor` already resolves mode + live-server reachability and exits 2 on structural error; env-driven deployments use it as their startup readiness check.

## Capabilities

### New Capabilities

- `setup-command`: The `scryrs setup <mode>` command surface — `local` and `live` modes, the config it writes (`scryrs.json` remote section, `.scryrs/` scaffold), required-vs-optional inputs, fail-fast validation, idempotency/merge semantics, and discoverability via help/help-json.

### Modified Capabilities

- `init-installer`: `scryrs init` is reduced to hook installation only. The mode selection, live-config validation, `scryrs.json` writing, live next-step text, and "installer may create scryrs.json in live mode" requirements are removed from `init` (relocated to `setup-command`). Hook embedding, project-local write targets, unsupported-harness handling, pre-existing-file refusal, Claude Code settings merge, and source-checkout self-install rules remain on `init`.
- `workspace-live-bootstrap`: Live bootstrap (`scryrs.json` writing, `.scryrs/compose.yml`, overrides-only `.scryrs/.env`, `scryrs up`) is triggered by `scryrs setup live` instead of `scryrs init --mode live`, and the compose/`docker_network` scaffold becomes an explicit opt-in rather than a mandatory part of live config.
- `cli-discovery-ux`: Help text and help-json document the new `setup` command and the reduced `init` surface (no `--mode`/live args on `init`), and the live-bootstrap flow becomes `scryrs init` → `scryrs setup live` → `scryrs up`.

> Relocation (not a spec delta here): the `add-init-interactive-wizard` change is **completed in code** — a TTY live-config wizard now lives on `scryrs init`. With this split that wizard belongs on `scryrs setup live`, so this change **migrates** the wizard out of `init` (which collects no config and must never prompt) into `setup live`. Its capability spec is not yet promoted (`add-init-interactive-wizard` is unarchived), so the retarget is enforced via the `setup-command` requirement "Interactive live-config collection belongs to `setup live`, not `init`" plus a migration task, rather than a Modified Capability delta. If `add-init-interactive-wizard` is archived before this change applies, add a `init-interactive-wizard` MODIFIED delta to retarget it formally (see design Open Questions).

## Impact

- **CLI surface (BREAKING)**: `scryrs init` loses `--mode` and all live-config flags; consumers and scripts that relied on `scryrs init --mode live ...` must switch to `scryrs init --agent <name>` followed by `scryrs setup live ...`.
- **Implementation scope**: `crates/scryrs-cli/src/init.rs` (slimmed), a new `setup` module, dispatch/help/help-json surfaces, golden snapshots, init/setup tests, README/CLI docs.
- **Coordination with `add-init-interactive-wizard`**: that change's wizard implementation must land on `setup live`; sequence so the wizard is not built against an `init` surface that this change removes.
- **Downstream (Swarm)**: the worker Dockerfile collapses to `scryrs init --agent pi` (hook only, no `--mode` trickery); the agent entrypoint adds a `scryrs doctor` readiness probe; the scryrs sidecar must be wired into the manager startup. (Tracked separately in the dev-swarm repo.)
- **No changes** to hook source (`hooks/pi/index.ts`, `hooks/claude-code/scryrs-hook.mjs`), TraceEvent schema, record transport, runtime config-resolution precedence (`CLI > env > .scryrs/.env > scryrs.json`), server endpoints, or `scryrs doctor`.
