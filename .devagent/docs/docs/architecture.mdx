# Architecture

scryrs is a Rust workspace split by product capability: the CLI composes feature-gated crates, the deterministic core owns trace and hotspot logic, guardrail crates bound risky behavior, and adapters publish knowledge to documentation systems without making any single docs framework mandatory.

The key architectural separation: **suite crates own product capabilities**, **guardrail crates bound unsafe or privacy-sensitive behavior**, and **adapters are publishing surfaces**. `crates/scryrs-cli` composes those pieces through Cargo features; `crates/scryrs-types` keeps cross-crate contracts small and explicit.

The LLM layer is not in the policy path. `crates/scryrs-llm` is an optional transport boundary for explanation, suggestion ranking, or patch drafting. Deterministic crates produce evidence first; model-backed features can interpret that evidence later.

## Crate Responsibilities

| Crate | Role | Why it exists |
| --- | --- | --- |
| `crates/scryrs-cli` | Main `scryrs` binary | Keeps user, CI, and agent command behavior in one feature-gated shell. |
| `crates/scryrs-types` | Shared contracts | Prevents circular dependencies and keeps cross-crate data shapes explicit. |
| `crates/scryrs-core` | Trace and hotspot foundation | Preserves standalone hotspot detection without docs-site assumptions. Exports shared `per_event_contribution()` contract for deterministic batch and live scoring alignment. |
| `crates/scryrs-server` | Central trace ingest and live hotspot server | Long-lived HTTP server for `POST /v1/trace-events/batch` with server-owned SQLite persistence and first-writer-wins idempotency. Also exposes read-only `GET /v1/repositories/{repository_id}/hotspots` (cumulative and session-scoped live hotspot rankings) and `GET /v1/repositories/{repository_id}/signals` (SSE stream of `HotspotSignal` records with `?after=<signal_id>` cursor replay/resume). Signals are fanned out via a `tokio::sync::broadcast` channel; subscribers receive a replay-then-live stream that prevents gaps. |
| `crates/scryrs-graph` | Knowledge graph foundation | Owns routing graph concepts separate from trace collection. |
| `crates/scryrs-curator` | Deterministic proposal engine | Turns hotspot and graph evidence into reviewable docs, skill, decision, memory, or grouping proposals. |
| `crates/scryrs-curator-llm` | Optional model-assisted curator layer | Drafts or suggests proposals over bounded evidence without entering deterministic ingest, scoring, graph, or route paths. |
| `crates/scryrs-runtime` | Agent-side routing foundation | Helps future agents choose relevant context faster. |
| `crates/scryrs-adapter-markdown` | Markdown publishing adapter | Supports generic docs repos and simple file-based output. |
| `crates/scryrs-adapter-rspress` | Rspress publishing adapter | Makes Rspress first-class without making it foundational. |
| `crates/scryrs-policy` | Deterministic guardrail decisions | Keeps policy decisions in code, not prompts. |
| `crates/scryrs-sandbox` | Tool and filesystem capability policy | Establishes read/write/exec boundaries before tool execution exists. |
| `crates/scryrs-telemetry` | Privacy and telemetry defaults | Makes observability opt-in and redaction-first. |
| `crates/scryrs-llm` | Optional model boundary | Keeps provider/model transport replaceable and outside deterministic policy. |
| `xtask` | Repository automation | Provides a stable place for bootstrap, CI, release, and generated-artifact tasks. |

## Cargo Feature Model

`crates/scryrs-cli/Cargo.toml` is the composition point.

```text
default
= core + graph + curator + markdown + runtime + guardrails + server

guardrails
= policy + sandbox + telemetry

full
= default + llm + rspress
```

This lets users keep the CLI lean for standalone hotspot analysis while enabling richer suite behavior when needed:

```bash
cargo run -p scryrs-cli -- hotspots .
cargo run -p scryrs-cli --no-default-features --features core -- hotspots .
cargo run -p scryrs-cli --features full -- hotspots .
```

## Source-of-Truth Rules

- `Cargo.toml` defines workspace membership, shared dependency aliases, and workspace lint policy.
- `crates/scryrs-cli/Cargo.toml` defines the product feature matrix.
- `crates/scryrs-types/src/lib.rs` owns shared contracts used across suite crates.
- `.devagent/architecture/` owns LikeC4 source for embedded architecture diagrams on this page.
- `scripts/check`, `scripts/test`, and `scripts/security` are local guardrail entry points.
- `.github/workflows/ci.yml` mirrors fast-lane checks for pull requests and `main`.
- `deny.toml`, `clippy.toml`, `rustfmt.toml`, and `rust-toolchain.toml` capture tool policy.

## Test Coverage & Verification

### CI lanes

- **Pull request**: `cargo test --workspace --all-targets --all-features --locked` (all Rust tests across workspace crates).
- **Hook verification**: A separate CI job in `.github/workflows/ci.yml` runs `scripts/hook-test` on every PR, checking Claude Code hook JSON shaping, fail-open, and transparency. Runtime: &lt;5 seconds, Node-only.
- **Cross-harness e2e** (`scripts/verify-trace-capture`): Docker-backed end-to-end verification exercising both Claude Code and Pi hook fixtures against a real `scryrs` binary. Measured runtime ~41 seconds (cached build). Assigned to **nightly lane** pending CI Docker-in-Docker support.

### Hook verification

- `scripts/hook-test` — Node-only Claude Code hook contract checks (JSON shaping, fail-open, transparency), runs on every PR.
- `scripts/verify-trace-capture` — Docker-backed cross-harness e2e (Claude Code + Pi fixtures against real `scryrs` binary).
- `scripts/verification/installed-hook-e2e.mjs` — End-to-end validation that `scryrs init --agent claude-code|pi` produces loadable, functional hook artifacts in a consumer project context.

### Local test lanes

- `scripts/test` — Rust tests only (default lane).
- `scripts/test --full` — Full suite: Rust tests followed by `scripts/hook-test`.

## Current Limitations

- `scryrs hotspots` is production-level: it opens `.scryrs/scryrs.db` via `TraceQuery`, runs deterministic scoring with a documented weight table and six-key tie-break, emits a versioned `HotspotsReport` (schemaVersion 1.0.0) to stdout and `.scryrs/hotspots.json`, and handles empty/missing/corrupt stores with correct exit codes. The `scryrs record`, `scryrs init`, and `scryrs server` commands are similarly production-grade with Docker-backed verification. The `scryrs server` command runs a long-lived HTTP ingest endpoint (`POST /v1/trace-events/batch`) with server-owned SQLite persistence and first-writer-wins idempotency.
  - **Live hotspot accumulators (v2 server schema)**: The server store maintains `hotspot_accumulators` and `hotspot_signals` tables. Accepted subject-bearing events trigger incremental accumulator updates in the same SQLite transaction as event inserts. A cumulative score crossing the configurable `signal_threshold` (default 10) emits an append-only `HotspotSignal` with subject identity, score delta, window tag, and timestamp-ordered evidence row IDs.
  - **Shared scoring contract**: Live accumulation reuses `per_event_contribution()` from `crates/scryrs-core/src/scoring.rs`, keeping deterministic batch scoring and live accumulation aligned on base weights and failure bonus without duplicated weight tables.
  - **Live hotspot query API**: `GET /v1/repositories/{repository_id}/hotspots` returns a versioned `LiveHotspotsResponse` with ranked `HotspotEntry` items. Supports `?window=cumulative` (default, cumulative scores from `hotspot_accumulators`) and optional `?session_id` (session-scoped recomputation via `score_hotspots` over filtered `server_trace_events`). Entries are ranked by the same six-key tie-break chain used by `scryrs hotspots`.
  - **SSE signal streaming**: `GET /v1/repositories/{repository_id}/signals` returns a `text/event-stream` with `HotspotSignal` records (id, subject identity, score, delta, window, threshold, evidence row IDs, created-at timestamp). Supports `?after=<signal_id>` for cursor-based replay/resume. The stream is infinite (replay phases then live broadcast); subscribers should disconnect cleanly rather than draining it.
  - **Graph / route / proposal state**: `scryrs graph`, `scryrs route`, and `scryrs propose` are shipped artifact commands. Current graph build is structural (hotspot subject nodes plus optional docs hierarchy), route generation is one-entry-per-node with explicit `contains` grouping only, and proposal generation is review-first JSON output under `.scryrs/proposals/` only.
  - **Optional LLM boundary**: `crates/scryrs-curator-llm` is library-only. It adds bounded drafting and grouping suggestion APIs, but no CLI command, no default-feature dependency, and no automatic mutation of graph, route, or docs truth.
  - **Still deferred**: runtime retrieval decisions, proposal acceptance workflow, and adapter publishing remain downstream work.
- LikeC4 views on this page are high-level architecture projections, not exhaustive Cargo dependency graphs.
- Rust toolchain is not installed in the current local environment, so cargo-backed verification is blocked until `rust-toolchain.toml` can be installed.
- `tests/golden/`, `tests/integration/`, and `tests/llm/` are `.gitkeep` placeholder directories with no active test suites. These remain deferred until a real test suite is designed to occupy each directory. Adding tests to these directories before their purpose is defined would create untethered suites with no clear assertion boundary.
