# Product Roadmap

scryrs should ship in strict order: first become a deterministic trace-capture CLI that never interferes with tool results, then turn captured evidence into hotspot outputs, then expand into graph, proposal, adapter, and runtime products.

## Current Starting Point

The Phase 1 deterministic proxy-capture boundary is implemented:

- `crates/scryrs-types/src/lib.rs` defines `TraceEvent` and related payload types.
- `crates/scryrs-core/src/lib.rs` provides JSONL event ingestion with semantic validation, rejection diagnostics, and an append-only `.scryrs/scryrs.db` SQLite store.
- `crates/scryrs-cli/src/lib.rs` exposes three commands: `scryrs record --stdin|--file <PATH>`, `scryrs hotspots <PATH>`, and `scryrs init --agent <NAME>` for installing reference hooks.
- Reference hooks live at `hooks/pi/` (TypeScript extension) and `hooks/claude-code/` (JavaScript PreToolUse hook).
- `scryrs.json` defines the hook-interface and record-invocation manifest at the repository root.
- Docker-backed fail-open verification (`scripts/verify-trace-capture`) exercises both hooks end-to-end against the real `scryrs` binary.

Phase 2 hotspot materialization is delivered: `scryrs hotspots <PATH>` opens `.scryrs/scryrs.db` via `TraceQuery`, runs deterministic scoring with a documented weight table and six-key tie-break, emits a versioned `HotspotsReport` (schemaVersion 1.0.0) to stdout and `.scryrs/hotspots.json`, and handles empty/missing/corrupt stores with correct exit codes. E2E tests prove the full record → SQLite → hotspots → artifact pipeline. Phase 4+ features (graph, proposal, adapter, runtime, MCP, LLM) remain deferred per the delivery-phase ordering below. Phase 3 (Dashboard) adds the local visual browsing surface on top of hotspot artifacts.

Key scope rule: **do not build later-suite features on placeholder ingest paths**. Every later phase depends on deterministic evidence collected by earlier phases.

## First Usable Product Boundary

First honest product cut is not "scryrs can publish docs." First honest cut is:

1. agent hook forwards raw events to scryrs,
2. `scryrs record` persists them,
3. `scryrs hotspots` materializes stable hotspot outputs,
4. scryrs proves it never rewrites underlying tool stdout, stderr, exit code, or semantics.

Until that exists, graph, proposal, and adapter work are downstream fiction.

## Delivery Phases

| Phase | Goal | Must exist before phase is done | Explicitly defer until later |
| --- | --- | --- | --- |
| 1. Deterministic proxy capture | Make scryrs usable as trace collector, not tool executor | `scryrs record`, local event store, `scryrs.json` hook manifest, reference Pi + Claude hooks, `scryrs init --agent`, fail-open verification | Graph building, docs generation, route explanation, LLM summaries |
| 2. Hotspot materialization | Turn persisted traces into stable CLI outputs and files | Real `scryrs hotspots`, deterministic aggregation rules, stable JSON contract, optional file outputs like `.scryrs/hotspots.json` | Proposal engine, docs adapters, runtime retrieval |
| 3. Dashboard | Make hotspot and session data visually browsable | `scryrs dashboard` CLI command, local HTTP server, embedded Vue.js SPA, hotspot table view, session timeline, event distribution chart, per-subject drill-down, extensible component architecture | Graph visualization, route exploration, hosted/multi-user dashboard, real-time updates, data mutation |
| 4. Graph and route manifests | Convert hotspots plus docs structure into machine-readable routing data | `scryrs graph build`, graph schema, route manifest schema, evidence links from hotspot output to graph nodes | Human-facing doc proposals, adapter-specific publishing polish |
| 5. Proposal engine | Produce reviewable docs / ADR / skill / memory proposals from evidence | `scryrs suggest-docs` or `scryrs propose`, proposal storage/inbox format, explicit review workflow | Silent auto-merge, autonomous doc rewrites |
| 6. Publishing adapters | Publish reviewed knowledge into docs systems without making one mandatory | Markdown adapter first, then Rspress adapter, export paths for `llms.txt` and nested manifests | Framework-specific lock-in, dashboard-first work |
| 7. Runtime retrieval | Help future agents load right context faster | `scryrs route explain`, route hints, retrieval-friendly exports, explainable reasons | Opaque ranking, hidden memory mutation |
| 8. Optional LLM layer | Add explanation and drafting help outside deterministic policy path | Bounded `scryrs-llm` usage for ranking, summarization, patch drafting only | Using models for policy, ingest validation, or trace truth |

## Phase 1 — Deterministic Proxy Capture

This phase is the implemented foundation that creates executable truth for scryrs' identity.

Completed deliverables:

- `scryrs record` CLI surface in `crates/scryrs-cli/src/lib.rs`.
- Ingest and persistence path in `crates/scryrs-core/src/ingestion.rs`, `crates/scryrs-core/src/store.rs`.
- Local append-only event store at `.scryrs/scryrs.db`.
- `scryrs.json` manifest describing hook interface, intercepted events, and `scryrs record` invocation.
- Reference hooks under `hooks/pi/` and `hooks/claude-code/`.
- `scryrs init --agent <name>` installer for deterministic hook setup.
- Docker-backed fail-open verification via `scripts/verify-trace-capture`.

Accepted limitations:

- Claude Code hook is PreToolUse-only (no lifecycle events).
- Pi hook captures `SessionStart` but not `SessionEnd` (shutdown-hook timing limitation).

Hard non-goals for this phase (still deferred):

- scryrs must not proxy tool execution itself.
- scryrs must not become an MCP server or callable business tool.
- hooks must not contain business logic beyond formatting plus subprocess delegation.

## Phase 2 - Hotspot Materialization ✅ Delivered

This phase turns raw event capture into first real product value.

Completed deliverables:

- `scryrs hotspots <PATH>` reads `.scryrs/scryrs.db` via `TraceQuery` in `crates/scryrs-core/src/scoring.rs`.
- Deterministic hotspot scoring with explicit weight table: `WEIGHT_FILE_OPENED=1`, `WEIGHT_SEARCH_RUN=2`, `WEIGHT_SYMBOL_INSPECTED=2`, `WEIGHT_COMMAND_EXECUTED=1`, `WEIGHT_DOC_RETRIEVED=2`, `WEIGHT_EDIT_MADE=3`, `WEIGHT_FAILED_LOOKUP=4`, with `FAILURE_BONUS=2` per-failure.
- Six-key tie-break chain: `score DESC, sessionCount DESC, lastSeen DESC, subjectKind ASC, subject ASC, firstEventId ASC`.
- Stable JSON contract: `HotspotsReport` (schemaVersion 1.0.0) with `runMetadata`, `entries`, and `command` fields emitted to stdout.
- `.scryrs/hotspots.json` artifact file written alongside stdout output.
- E2E test (`crates/scryrs-cli/tests/hotspot_e2e.rs`) covers full record → SQLite → hotspots → artifact pipeline, multievent-family fixtures, empty-store, and missing-store coverage.

Release boundary (achieved):

```text
scryrs record --stdin
scryrs hotspots <PATH>
/.scryrs/hotspots.json
```

Phase 2 is strong — scryrs already has credible standalone utility.

## Phase 3 - Dashboard

Only start this after hotspots stabilize. The dashboard is the first visual product surface and establishes the `scryrs dashboard` CLI command and frontend architecture before graph/route data multiplies the visualization surface later.

Required deliverables:

- `scryrs dashboard [--port <PORT>] [--bind <ADDR>] [--no-open] [--dev]` CLI command that starts a local HTTP server and opens a Vue.js SPA in the default browser.
- Local REST API serving `GET /api/hotspots` (from `.scryrs/hotspots.json`), `GET /api/sessions`, and `GET /api/events` (from `.scryrs/scryrs.db`).
- Embedded Vue.js SPA built with Vue 3 + Vite + strict TypeScript + Bun + Tailwind CSS v4 + shadcn-vue + Reka UI + lucide-vue + Pinia + Vue Router, compiled into the binary via `rust-embed`.
- Hotspot report table view (sortable, ranked entries with score, session count, timestamps).
- Per-subject drill-down view with event-type breakdown and session timeline.
- Session list and detail views with scrollable event lists.
- Event distribution visualization grouped by event type, filterable by session.
- Component architecture with extensible slots for future graph/route views.
- Embedded SPA as static assets in the Rust binary (single-file deploy).
- New crate `crates/scryrs-dashboard/` with `axum` + `tokio` HTTP server.

Accepted limitations:

- No graph visualization or route exploration — those belong in Phase 4+ when the data exists.
- No hosted or multi-user dashboard — this is local-only.
- No authentication, no write API, no data mutation through the dashboard.
- No real-time or streaming updates — the dashboard reads current artifact files on each API request.
- SPA assets are embedded into the binary (acceptable for single-binary CLI tool).

## Phase 4 - Graph and Route Manifests

Only start this after dashboard and hotspot outputs stabilize. Otherwise graph schema will chase unstable evidence shapes.

Required deliverables:

- Build graph nodes and relationships from docs structure plus hotspot evidence.
- Emit stable graph and route artifacts such as `/.scryrs/graph.json` and `/.scryrs/routes.json`.
- Preserve explainability: every routing hint should cite hotspot or document evidence.

This phase is where scryrs stops being only a recorder and starts becoming routing infrastructure.

## Phase 5 - Proposal Engine

This phase turns evidence into reviewable change suggestions.

Required deliverables:

- Proposal types for docs notes, ADRs, debugging playbooks, skills, and memory patches.
- Review queue or inbox semantics instead of silent writes.
- Deterministic templates that cite graph and hotspot evidence.

Do not skip straight to automatic doc mutation. That would destroy trust.

## Phase 6 - Publishing Adapters

Adapters come after proposals because scryrs should publish approved knowledge, not invent source-of-truth content on its own.

Recommended order:

1. `crates/scryrs-adapter-markdown`
2. `crates/scryrs-adapter-rspress`
3. Additional framework adapters only after generic surfaces work

Why: Markdown keeps product core honest. If adapter logic only works when tied to one docs framework, architecture has already drifted.

## Phase 7 - Runtime Retrieval

Runtime work should consume prior artifacts, not bypass them.

Required deliverables:

- Explainable `scryrs route explain ...` behavior.
- Retrieval-friendly route manifests.
- Stable reason strings pointing back to graph links, hotspot evidence, and approved docs outputs.

This phase is when future agents should start reading less junk and loading more relevant context.

## Phase 8 - Optional LLM Layer

Model-backed features belong last because they are interpretation features, not truth-producing features.

Valid later uses:

- rank competing proposal drafts,
- summarize hotspot clusters,
- draft prose around already-determined evidence,
- explain route decisions in better human language.

Invalid uses:

- deciding what events mean during ingest,
- replacing deterministic hotspot scoring,
- making guardrail or policy decisions,
- silently mutating approved knowledge.

## Scope Guardrails

If roadmap starts to sprawl, cut work using these rules:

1. **No adapter-first work before real hotspots exist.**
2. **No graph-first work before persisted event evidence exists.**
3. **No runtime routing before graph outputs are stable.**
4. **No LLM dependency in ingestion, policy, or scoring path.**
5. **No dashboard or hosted analytics product before standalone CLI is useful.** ← satisfied as of Phase 2 delivery. The Phase 3 Dashboard is a local CLI dashboard, not a hosted analytics product. Hosted analytics remain deferred.
6. **No hook implementation that rewrites tool outputs or agent semantics.**

| --- | --- |
| M1 - Proxy capture alpha | Install reference hook, record traces, verify fail-open non-interference |
| M2 - Standalone hotspots beta | Record traces across sessions and produce real hotspot output from CLI |
| M3 - Local dashboard beta | Start `scryrs dashboard`, browse hotspots, sessions, and events in a browser |
| M4 - Routing data beta | Export graph and route manifests for downstream consumers |
| M5 - Proposal loop beta | Generate reviewable docs and memory proposals from evidence |
| M6 - Adapter suite beta | Publish approved knowledge into Markdown and Rspress surfaces |
